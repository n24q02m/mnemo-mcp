## 2024-03-08 - [UnboundLocalError due to Shadowed Imports in Error Branches]
**Learning:** In Python, if a module is imported inside a nested block (like a `try/except` inside a `for` loop), it binds the module name as a *local variable* in the entire function scope. If that code block doesn't execute (e.g., when passing a `list` to a function that expects strings, but attempts a string parse anyway and fails, or skips the block entirely), subsequent code referencing the module will fail with an `UnboundLocalError: cannot access local variable 'json' where it is not associated with a value`, even if there's a global import of the same module at the top of the file. This hidden behavior completely broke a performance optimization where `db.import_jsonl` was meant to accept a raw `list` to bypass redundant serialization, because the `import json` was in the `str` parsing branch.
**Action:** When optimizing functions to support raw data types (lists/dicts) directly instead of serializing strings, always ensure that no nested imports inside parsing logic shadow global module names. It is best to stick to top-level module imports unless lazy loading is strictly required, in which case the lazy import should be confined to its immediate usage scope or handled carefully across all function branches.
