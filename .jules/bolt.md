## 2025-03-24 - N+1 Query Optimization in Entity Upserts
**Learning:** For small-to-medium datasets (e.g. graph entity extraction returning 10-200 entities), Python sqlite3 driver's looping `execute` for point reads (using a `SELECT` against a unique index) followed by a bulk `executemany` for inserts and updates is incredibly fast, even faster than constructing massive `IN (VALUES ...)` queries or relying on `executemany` UPSERTs without `RETURNING`.
**Action:** Replaced the N+1 looping `execute(SELECT)` and conditional `execute(INSERT)`/`execute(UPDATE)` statements in `graph.py`'s `upsert_entities` function with an initial loop to deduplicate unique entities and query the DB, followed by unified `conn.executemany` bulk insertions and updates. This delivered measurable ~2x improvements on inserts/updates without relying on external dependencies or risking query size limits.

## 2025-04-05 - HTTP Client Caching Overhead
**Learning:** In Python, instantiating high-level HTTP and SDK clients (like `httpx.Client`, `openai.OpenAI`, or `cohere.ClientV2`) incurs a massive initialization overhead. When performed inside loops or repeated function calls, this forces redundant parsing of configuration, header construction, and prevents HTTP connection pooling (missing out on TCP/TLS reuse). In benchmarking against `openai.OpenAI`, caching the client improved execution time by over 100x compared to recreating it per request.
**Action:** Implemented caching for HTTP/SDK clients inside the `CloudEmbeddingBackend` class. The clients are now stored as instance variables (`_jina_client`, `_gemini_client`, `_openai_client`, `_cohere_client`) and are lazily initialized upon the first call, allowing subsequent embedding calls to reuse the underlying connection pools and bypass the instantiation tax.
